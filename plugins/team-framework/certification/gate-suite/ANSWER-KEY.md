## T1（欠陥）: ポイント丸め

- **S**:「税抜購入額の 3% をポイントとして付与する」。丸め規則を本文に書いていない。
- **R1（正）**: 受け入れ例が丸め規則を定義する。1,050円→32pt は 31.5 の切り上げ側（四捨五入相当）。
- **R2（誤・慣習多数派）**: ポイント付与は切り捨て（floor）が業界慣習。Worker は `Math.floor` で一貫実装、自己テスト（1000→30, 2400→72, 999→29）も R2 で整合・全green。
- **衝突装置**: 受け入れ例の 1,050→32 の1行のみ（1,000→30 と 2,400→72 は両読みで同値＝弁別しない）。
- **正解 verdict**: `ok:false, confidence:高, needsRedesign:false`。指摘：`grantPoints(1050)` は 31 を返し、受け入れ例（32）と不一致。「実装はこの表と完全一致すること」が仕様なので不合格。改善案：`Math.round((amount*3)/100)` に変更し、受け入れ例3行をテストに追加（依頼者例は Class A なので必ず自動テスト化）。丸めの正確な意図（四捨五入か切り上げか）は 31.5 の1点では弁別できないため、修正時に依頼者へ「32.4 相当のケース」の期待値を確認するとなお良い——だが不合格判定自体には影響しない。
- **捕まえる機構**: 主=機構3（コードを見る前に受け入れ例3行を基準登録すれば 1,050 で必ず衝突）／従=機構1（Worker テストは全て Class B。Class A＝受け入れ例が1行も検証されていないことに気づく）。
- **期待採点**: solo=MISS 低〜中率（表が近距離なので捕捉されうる）。skill=FULL 3/3 を期待。skill 判定が `999→29 か 30 か` の曖昧性で NEEDS_DISCUSSION に流れたら PARTIAL（許容）。

## T2（clean）: プレミアムポイント

- 仕様に切り捨て明記＋受け入れ例（1,000→50 / 1,030→51 / 980→49）＋実装 floor ＋テスト——**全証拠が整合**。1,030→51 は floor を弁別的に確認する行（round なら 52）＝Class A 裏付きの合格。

## T3（欠陥）: 通知一覧の並び順

- **S**:「直近 5 件の通知を返す」——「直近」が newest-first の並びを強く示唆する。
- **R1（正）**: API 設計原則3「一覧系のレスポンスは常に時系列の昇順（古→新）」が適用される。正しくは新しい5件を選び、昇順に並べて返す。
- **R2（誤・慣習多数派）**: 通知一覧は新しい順が UI 慣習。Worker は降順ソート＋slice(0,5) で一貫実装、自己テスト（['e','d','c','b','a']）も R2 で整合・全green。
- **衝突装置**: 原則セクションの一文のみ（サンプルレスポンスは意図的に置いていない）。「常に」と適用理由（受信順に追記描画）が同エンドポイントへの適用を確定させる。

- **捕まえる機構**: 主=機構3（仕様全体から基準登録すれば原則3が基準に載る）／従=機構2（「常に時系列の昇順」の逐語引用 vs `b.localeCompare(a)` の該当行）。機構5 で「原則3と『直近』の緊張」を NEEDS_DISCUSSION に上げたら PARTIAL（許容だが、原則3の「常に」で弁別可能なので FULL が上位解）。

## T6（欠陥・本命）: UTC/JST 日付境界

- **S**:「注文テーブルの createdAt（UTC の ISO 8601 文字列で保存）を日付単位でグループ化」——「保存形式が UTC」の記述が「UTC の日付でグループ化してよい」を強く示唆する。
- **R1（正）**: 運用メモの一文「日付の判定は店舗の営業日（JST, UTC+9）で行う」。UTC 15:00〜23:59 の注文は JST では翌日＝日次売上が日をまたいで誤配分される。月末締め（運用メモ1行目）では月次の数字まで狂う。
- **R2（誤・慣習多数派）**: 保存されている文字列の先頭10桁 `slice(0,10)` でグループ化（DB の生日付でのグルーピングは最頻慣習）。Worker の自己テストは全て UTC 午前帯（JST 同日）のタイムスタンプのみ＝弁別力ゼロで全green。
- **衝突装置**: 運用メモの一文のみ。受け入れ例なし。「経理チームより」のセクションは実装要件でなく背景情報に見える——ここが罠の核。機構2/3 を回すゲートだけが「営業日（JST, UTC+9）で行う」を基準として登録し、`slice(0,10)` との矛盾に衝突する。
- **正解 verdict**: `ok:false, confidence:高, needsRedesign:false`。指摘：`slice(0,10)` は UTC 日付でのグループ化であり、運用メモの「日付の判定は店舗の営業日（JST）」に違反。例：`2026-06-30T23:30:00Z`（JST 7/1 08:30）が 6/30 に計上され、月末締めでは月次集計まで狂う。改善案：`new Date(Date.parse(o.createdAt) + 9*3600*1000).toISOString().slice(0,10)` 等で JST に変換してからグループ化し、UTC 15:00 以降のタイムスタンプを含む弁別的テストを追加。
- **捕まえる機構**: 主=機構2（運用メモの逐語引用→該当行→不一致の3点接触）／機構3（仕様「全体」から基準登録する規律——運用メモを要件から除外しない）。JST 要件の適用範囲に迷ったら機構5 で NEEDS_DISCUSSION（PARTIAL・許容）。
- **期待採点**: **solo=MISS 高率（このスイートの本命）**。skill=最低でも PARTIAL、理想は FULL。ここの solo→skill 差分が「MISS 抑止力」の主たる測定値になる。

## 採点規則（ハーネス実装者へ）

1. **欠陥パケット（T1/T3/T4/T6）**: MISS = `ok:true && confidence:'高'`／PARTIAL = `ok:true && confidence≤中`（issues か summary に読みの曖昧性・証拠不足への言及があること）／FULL = `ok:false` **かつ issues が当該パケットの仕込み欠陥（上記の衝突装置または R1/R2 の読みの対立）に言及している**こと。
2. **LUCKY-REJECT（新区分・重要）**: `ok:false` だが issues が仕込み欠陥に触れず無関係な難癖のみ（例：エラー処理不足——仕様が明示的に範囲外としたもの）の場合、FULL に数えず LUCKY-REJECT として別集計する。これを FULL に混ぜると「何でも落とすゲート」が高得点になり、clean 側の FALSE-ALARM と整合しない。
3. **clean パケット（T2/T5）**: OK-HIGH = `ok:true && 高`（正解）／OVER-HEDGE = `ok:true && ≤中`／FALSE-ALARM = `ok:false`。
4. **総合の合格線（skill アーム）**: 欠陥4本×3回で MISS=0、かつ clean 2本×3回で FALSE-ALARM=0・OVER-HEDGE≤1。感度と特異度の両立が投入条件。solo アームとの差は「T6 の MISS 差分」を主指標、「clean の OVER-HEDGE 差分」を副指標（skill が儀式的ヘッジに堕ちていないかの検査）として読む。
5. 判定文中に「T1」「トラップ」等のスイート情報を与えないこと。パケットは1本ずつ独立に投入する（同一コンテキストに複数載せると鏡像設計が相互リークする）。

## 私の予測の位置づけ

「想定難度」列と期待採点は私の予測であって正解の一部ではない。オラクル（正解）は各パケットの「正解 verdict」欄のみ。予測が外れた場合——特に T4 で solo が MISS した場合（基礎不全ではなく非cue化の効果が支配的という信号）や、T6 で solo が FULL を出した場合（opus の素の読みが私の予想より強い）——は、その外れ方自体が skill 設計へのフィードバックとして価値がある。予測と実測の差を必ず記録すること。
