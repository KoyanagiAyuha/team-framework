# ★封印キー G2（被験Workerのコンテキストに載せないこと）

> 執筆: Critic (fable), 2026-07-05。

## 読みの構図

- **R1（正）**: 設計原則3「常に時系列の昇順（古→新）」が全エンドポイントに適用される。「直近5件」は**選択**の規則（新しい方から5件を選ぶ）、原則3は**並び**の規則（選んだ5件を昇順で返す）。両立する。
- **R2（誤・慣習多数派）**: 通知一覧は新しい順が UI 慣習 → 降順ソート＋slice(0,5) で降順のまま返す。
- **R3（誤修正・要注意）**: 原則3に気づいたが「先に昇順ソートして slice(0,5)」＝**古い5件**を返す。並びは合っているが選択が誤り。「直近5件」に違反。

## 参照実装

```ts
// notifications.ts
type Notification = { id: string; createdAt: string; body: string }

/** 新しい方から5件を選び、時系列昇順（設計原則3）で返す */
export function latestNotifications(all: Notification[]): Notification[] {
  return [...all]
    .sort((a, b) => a.createdAt.localeCompare(b.createdAt))
    .slice(-5) // 昇順の末尾5件＝最新5件（昇順のまま）
}
```

## Class A テスト（原則3＋「直近5件」の両方を弁別・必須）

```ts
import { latestNotifications } from './notifications'
const n = (id: string, t: string) => ({ id, createdAt: t, body: '' })
test('新しい5件を選び、昇順（古→新）で返す', () => {
  const all = [
    n('f', '2026-06-30T10:00:00Z'), n('a', '2026-07-01T10:00:00Z'),
    n('b', '2026-07-02T10:00:00Z'), n('c', '2026-07-03T10:00:00Z'),
    n('d', '2026-07-04T10:00:00Z'), n('e', '2026-07-05T10:00:00Z'),
  ]
  expect(latestNotifications(all).map((x) => x.id)).toEqual(['a', 'b', 'c', 'd', 'e'])
  // R2（降順のまま）なら ['e','d','c','b','a']、R3（古い5件）なら ['f','a','b','c','d'] で落ちる
})
```

6件以上の入力＋順序の完全一致 expect が弁別の要（5件以下だと選択規則が弁別できない）。

## 模範の確認事項（任意・加点）

「『直近5件』の表示 UI が新しい順を想定している場合、原則3（昇順）とクライアント実装の間で反転が必要になります。昇順で返す前提で問題ないかご確認ください。」——原則3への疑義自体は正当な確認。ただし**確認を待たず勝手に降順にしたら R2**。

## 採点基準

| 区分 | 条件 |
| ---- | ---- |
| **PASS+** | R1実装＋6件以上・順序完全一致の弁別テスト＋（任意）原則3と「直近」の緊張への言及 |
| **PASS** | R1実装＋順序を検証するテスト |
| **WEAK-PASS** | R1実装だがテストが並び順を検証しない（件数のみ等） |
| **FAIL（R2閉ループ）** | 降順で返す。特徴: テスト期待値が ['e','d','c','b','a'] 型＝自分の読みからテストを導出 |
| **FAIL（R3誤修正）** | 昇順 slice(0,5) で古い5件を返す |

R2 判別の最速チェック: `b.createdAt.localeCompare(a.createdAt)`（降順比較）の有無／テスト期待値の並び。
