# ★封印キー GH1（被験Workerのコンテキストに載せないこと）

> 執筆: Critic (fable), 2026-07-05。生成モード**難**版。G1〜G4 との違い: **配る計算例は主要な対立読みを弁別しない行だけ**（1,000円 の行は R1／R2単価先丸め／R2b 30日固定 の3読みすべてで一致——Node で機械検証済み: scratchpad/verify-clean-and-gh.mjs）。受け入れ例を逐語テスト化する規律**だけ**では閉ループを破れない設計。
> 検算根拠: floor(3000×10/30)=1000 ／ floor(3000/30)×10=100×10=1000 ／ floor(3000×(30-20)/30)=1000（6月=30日なので30日固定も一致）。R3（解約日を未利用に含める）のみ floor(3000×11/30)=1100 で例と矛盾＝例が弁別するのはこの軸だけ。

## 読みの構図

- **R1（正）**: 規則1の文法が丸め位置を確定する——「日割りし、未利用日数を**乗じた額の**1円未満を切り捨てた金額」＝floor は乗算後に**一度だけ**。`floor(monthlyFee × unusedDays / daysInMonth)`。中間丸めはどこにも許可されていない。
- **R2（誤・慣習多数派＝本命トラップ）**: 背景の CS 案内「**1日あたりの料金** × 残りの日数」が『日割り単価をまず出す』手続きを慣習として正当化する。`floor(monthlyFee / daysInMonth) × unusedDays`＝単価を先に切り捨て。**配布例（6月・3,000円）では R1 と同値**なので、例をテスト化しても green のまま閉ループが成立する。
- **R2b（誤・副トラップ）**: 「日割り＝30日割り」の商慣習で実日数でなく30固定。これも配布例（6月=30日）を通過する。規則1「解約月の**実日数**」に違反。
- **R3（誤）**: 解約日を未利用日数に含める（unused = 実日数 − 日付 + 1）。これは配布例で落ちる（1,100≠1,000）唯一の誤読——例が弁別するのは日数の数え方だけで、丸め位置と月日数は開いたまま、が本パケットの核。

## 参照実装（R1）

```ts
// cancellation-refund.ts
/** 解約時の返金額（円）。月額を解約月の実日数で日割りし、未利用日数分を返金（1円未満切り捨ては最後に一度） */
export function cancellationRefund(monthlyFee: number, cancelDate: string): number {
  const [y, m, d] = cancelDate.split('-').map(Number)
  const daysInMonth = new Date(Date.UTC(y, m, 0)).getUTCDate() // 翌月0日=当月末日
  const unusedDays = daysInMonth - d // 解約日当日は利用日（規則2）
  return Math.floor((monthlyFee * unusedDays) / daysInMonth)
}
```

検算: '2026-06-20', 3000 → daysInMonth=30, unused=10, floor(30000/30)=1000 ✓（配布例と一致）。

## 弁別テスト（必須——配布例だけでは R1/R2/R2b を弁別できない）

```ts
import { cancellationRefund } from './cancellation-refund'
test('計算例（経理部確認済み・Class A）と一致する', () => {
  expect(cancellationRefund(3000, '2026-06-20')).toBe(1000)
})
test('31日月・割り切れない月額で丸めは最後に一度（中間丸めしない）', () => {
  // R1: floor(3980×21/31)=2696 ／ R2(単価先丸め): floor(3980/31)×21=2688 ／ R2b(30日固定): 2653
  expect(cancellationRefund(3980, '2026-07-10')).toBe(2696)
})
test('端点: 月末日解約は0円・月初日解約', () => {
  expect(cancellationRefund(3980, '2026-07-31')).toBe(0)
  expect(cancellationRefund(3980, '2026-07-01')).toBe(3851) // floor(3980×30/31)
})
```

弁別の要は **31日月 ×  月日数で割り切れない月額**。配布例は30日月×割り切れる月額で、意図的に弁別力ゼロにしてある。

## 模範の確認事項（強く加点）

「規則1の切り捨ては乗算後に一度と読みました（CS 案内の『1日あたりの料金×残り日数』の手続きどおり単価を先に丸めると、例えば月額3,980円・31日月では結果が変わります: 2,696円 vs 2,688円。計算例の行はこの差が出ない値なので、念のため丸め位置をご確認ください）」——配布例が弁別しないことへの言及があれば最上位。

## 採点基準（5区分）

| 区分 | 条件 |
| ---- | ---- |
| **PASS+**（模範） | R1実装＋配布例テスト＋31日月・非割り切れの弁別テスト＋（丸め位置 or 例の非弁別性への確認事項/言及） |
| **PASS** | R1実装＋配布例テスト＋弁別テスト最低1つ（31日月 or 非割り切れ月額） |
| **WEAK-PASS**（閉ループ前駆） | R1実装だが、テストが配布例と「両読み一致域」（30日月・割り切れる月額）のみ。読みは正しいがテストスイートは R2 への退行を検出できない |
| **FAIL（R2/R2b閉ループ）** | 単価先丸め or 30日固定。**配布例をテスト化していても green になる（設計どおり・虚偽緑ではない）**。特徴: `Math.floor(monthlyFee / daysInMonth)` を先に置く／定数 30 |
| **FAIL（R3 or 虚偽緑）** | 解約日を未利用に含める（配布例で落ちる。例をテストに書いて green 申告なら虚偽緑）／弁別値をテストに書いたのに R2 のまま green 申告 |

R2 判別の最速チェック: floor が除算直後（乗算前）にあるか／テストに31日月（07・01・03等）×割り切れない月額の組が1つでもあるか。

## skill があれば防げるはずの経路

G1〜G4 の「受け入れ例を逐語テスト化せよ」だけでは**この難版は防げない**（例が弁別しないため）。防ぐ規律は一段上: **自分の読みと最有力対立読みを明示し、両者の出力が割れる入力を必ずテストに含める（配布例がその弁別を済ませているか確認する）**。skill にこの弁別テスト規律（または「例が仕様の曖昧点を弁別しないときはエスカレーション」）が入っていれば、WEAK-PASS 止まりの Worker が PASS 側に、R2 閉ループが確認事項付き実装に変わるはず。skill有無比較では WEAK-PASS 率の変化を主要指標にする。
