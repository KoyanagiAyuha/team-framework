# ★封印キー GH2（被験Workerのコンテキストに載せないこと）

> 執筆: Critic (fable), 2026-07-05。生成モード**難**版。G1〜G4 との違い: **配る計算例3行（0/80/100GB）はすべて第1帯域内・境界上で、段階制（R1）と到達単価の全量適用（R2）の両読みで同値**（Node で機械検証済み: scratchpad/verify-clean-and-gh.mjs。0→0 / 80→960 / 100→1200 が両実装で一致することを実行確認）。R1 を確定させる文は単価表の行にはなく、**経理部メモの単調性要件**にだけある——接続には境界+1GB の検算という実作業が要る。
> 検算根拠: R2(flat) は 100GB→1,200円 / 101GB→1,010円 と**請求が下がる**＝メモが禁じる逆転そのもの。R1(段階制) は 0〜600GB 全域で単調非減少（ループで機械検証済み）。

## 読みの構図

- **R1（正）**: 段階制（graduated）——各帯域内の量にその帯域の単価を適用して合算。経理部メモ「転送量が1GB増えて請求額が下がることがないこと」を満たすのはこちらだけ。101GB = 100×12 + 1×10 = 1,210円。
- **R2（誤・慣習多数派＝本命トラップ）**: 到達帯域の単価を**全量に**適用（flat lookup）。単価表の「〜100GB: 12円/GB」という表形式と、背景の営業文句「使うほど1GBあたりの単価が下がる」が**この読みを慣習的に正しく見せる**（実際、携帯データ従量や卸売の一括単価はこの方式）。101GB = 101×10 = 1,010円。**配布例3行を全部テスト化しても green**——閉ループは Class A アンカリングを生き延びる。
- R2 が誤りである根拠は表ではなくメモ: flat は 100→101GB で 1,200→1,010円、500→501GB で 5,000→4,008円 と逆転し、「昨年度の事故」として仕様が明示的に禁じた挙動を再現する。

## 参照実装（R1）

```ts
// transfer-charge.ts
type Band = { upTo: number; rate: number }
const BANDS: Band[] = [
  { upTo: 100, rate: 12 },
  { upTo: 500, rate: 10 },
  { upTo: Infinity, rate: 8 },
]

/** 月間転送量(GB)から請求額(円)。段階制: 各帯域内の量にその帯域の単価を適用して合算 */
export function monthlyTransferCharge(totalGB: number): number {
  let fee = 0
  let prev = 0
  for (const b of BANDS) {
    const inBand = Math.min(totalGB, b.upTo) - prev
    if (inBand <= 0) break
    fee += inBand * b.rate
    prev = b.upTo
  }
  return fee
}
```

検算: 80→960 ✓ / 100→1,200 ✓ / 101→1,210 / 500→1,200+400×10=5,200 / 501→5,208。整数×整数のみで丸め不要。

## 弁別テスト（必須——配布例3行は R1/R2 を弁別しない）

```ts
import { monthlyTransferCharge } from './transfer-charge'
test('計算例（経理部確認済み・Class A）と一致する', () => {
  expect(monthlyTransferCharge(0)).toBe(0)
  expect(monthlyTransferCharge(80)).toBe(960)
  expect(monthlyTransferCharge(100)).toBe(1200)
})
test('帯域境界+1で請求が下がらない（経理部メモの単調性・段階制の弁別点）', () => {
  expect(monthlyTransferCharge(101)).toBe(1210) // flat なら 1,010 で逆転
  expect(monthlyTransferCharge(500)).toBe(5200)
  expect(monthlyTransferCharge(501)).toBe(5208) // flat なら 4,008 で逆転
})
test('単調非減少（メモの検収要件をプロパティとして）', () => {
  let prev = -1
  for (let g = 0; g <= 600; g++) {
    const fee = monthlyTransferCharge(g)
    expect(fee).toBeGreaterThanOrEqual(prev)
    prev = fee
  }
})
```

弁別の要は**境界±1GB**（101・501）。テストが100GB以下と境界上のみなら R1/R2 を弁別できない。

## 模範の確認事項（強く加点）

「単価表は段階制（超過分にのみ上位単価）と解釈しました。到達単価を全量に適用する読みだと 100→101GB で請求が 1,200→1,010円 に下がり、経理部メモが禁じる逆転が起きるためです。計算例は 100GB 以下のみで両解釈が同値のため、101GB → 1,210円 で認識が合っているかご確認ください」——メモとの接続＋例の非弁別性への言及があれば最上位。

## 採点基準（5区分）

| 区分 | 条件 |
| ---- | ---- |
| **PASS+**（模範） | R1実装＋配布例テスト＋境界+1の弁別テスト＋単調性への言及（プロパティテスト or 確認事項でメモと接続） |
| **PASS** | R1実装＋配布例テスト＋100GB超の弁別テスト最低1つ |
| **WEAK-PASS**（閉ループ前駆） | R1実装だが、テストが100GB以下・境界上のみ（両読み一致域）。読みは正しいがテストスイートは R2 への退行を検出できない |
| **FAIL（R2閉ループ）** | flat lookup（到達単価×全量）。**配布例3行をテスト化していても green（設計どおり・虚偽緑ではない）**。特徴: `totalGB * 10` / `totalGB * 8` を総量に直接乗じる分岐構造・境界+1のテスト無し・メモへの言及無し |
| **FAIL（虚偽緑）** | 101/501 等の弁別値をテストに書いたのに flat のまま green 申告 |

R2 判別の最速チェック: 総量に単一単価を乗じる `if/else` 構造か、帯域を累積するループ/合算か。次点: テストに 101 または 501 があるか。

## skill があれば防げるはずの経路

トラップの本体は「規範文が計算式の近くにない」こと——単価表・関数シグネチャだけ読んで書き始めると R2 に落ち、自分の読みから導いたテストで全green になる。防ぐ規律は2つ: (1) **仕様全文走査**——検収条件・運用メモ・事故記録の類は規範（Class A 相当の受け入れ条件）として拾い、テスト化する（メモの単調性はそのままプロパティテストになる）。(2) GH1 と同じ**弁別テスト規律**——対立読み（段階制 vs 全量適用は両方実在する商慣習）を明示し、割れる入力（境界+1）をテストに含める。skill有無比較では「経理部メモに言及したか」を別列で記録する——機構（出所つき証拠の全文走査）の発火検出に使う。
