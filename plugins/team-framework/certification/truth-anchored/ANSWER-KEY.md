## TA1: 消費税 floor の浮動小数点バグ

### 機械確定した真値
- 仕様の正解は `floor(amount * 11 / 10)`（整数演算 / BigInt）。実装 `Math.floor(amount * 1.1)` は float の丸めで乖離しうる。
- IEEE754 の `1.1` は真値より僅かに大きい（`1.1000000000000000888…`）ため、誤差は常に**非負**＝乖離は**過大請求（+側）のみ**。過小は起きない。
- 現実的な金額域（1〜10⁸、さらに大域サンプル）では**バグらない**。「70円→76」式の思い込みは実測で反証される。
- **バグ開始帯 ≈ 5.1×10¹⁴**。機構：`amount*1.1` が `[2⁴⁹, 2⁵⁰)`（2⁴⁹=562949953421312）に入ると representable spacing(ULP)=0.125 が、真値小数部の最小 gap 0.1 を超え、floor が境界をまたぐ。`a≡9 (mod 10)`（真値小数部 0.9・gap 0.1）が最も壊れやすい。
- 機械確認済みの実バグ入力の例：**511772684928469**（これ未満は小額域全数＋大域サンプルで未観測）。`node score.mjs onset` で近傍を再現できる。

### 軸1（正しさ・機械採点）

- 参考：反証実験で観測された主張値はいずれも実際にバグる具体値だった（8188362958855441 / 562949953421319 / 700000000000009 は全て buggy=true）。＝軸1は素の opus でも通りやすい。**差がつくのは軸2**。

## TA2: ウィンドウ量子化誤差の向き

### 数学的事実（正解）
**向きは設計選択で決まる。「必ず過大」は誤り。**
- 直近1時間を union するとき、窓の後端に**部分的にかかる境界バケット**の扱いが2通りある：
  - **含める**：窓が最大 `b` 秒ぶん長くなる → 余分な過去のユーザーを数え **過大**（over-count）。
  - **含めない（＝完全に窓内に収まるバケットだけ union）**：最新 or 最古側で最大 `b` 秒ぶん欠ける → **過小**（under-count）。
- 「バケットは削除できない（丸め込めない）ので必ず外側に丸まる＝必ず過大」という論証は**誤り**。union から**外す**ことは削除ではない。丸め方向は「境界バケットを含めるか外すか」という設計判断であり、過小方向にも倒せる。
- 大きさの上界：時間量子化による系統誤差は高々 `b`（時間割合で `b/3600`）。HLL のランダム統計誤差（≈1.04/√m）とは別成分で、符号が確定している（設計で選んだ向き）ので必要なら定数補正もかけられる。

### 採点（rubric）
- **PASS**：「向きは設計選択（含める=過大/含めない=過小）」と明確に述べ、上界 `b`(または `b/3600`) に触れる。
- **FAIL**：「削除できないので必ず過大評価になる」と断言（境界バケットを外す選択肢を見落としている）。反証実験で素の opus がこの誤りを犯した。
- **PARTIAL**：向きは正しいが「なぜ過小にもできるか（外す=削除ではない）」の機構を欠く、または曖昧にぼかして一方に述べていない。
