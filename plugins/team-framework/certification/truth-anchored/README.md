# truth-anchored 着任試験（judge不要・機械採点）

## なぜ別スイートか——circularity を構造的に切る

既存の着任試験（`gate-suite` / `generation-mode`）は最終的に**人間または judge モデルの読み**で採点する。judge がある限り、被験と同系モデルを judge に使うと**自己選好バイアス**が乗り、「skill が書式を教える → judge が収束と誤認」という偽陽性が混入する（実測：`tmp/pair-corpus/divergence-results-v2.md` の Control A〜C）。

このスイートは**正解が機械で検証できるタスクだけ**を集める。採点に judge を使わない＝

- **circularity が構造的に切れる**：被験の出した数値・境界を実コードで走らせ、真値と照合するだけ。judge の解釈が入らない。
- **skill の文体アーティファクトが効かない**：正しい/間違っているは実行が決める。fable 調に書こうが素朴に書こうが、数値が合わなければ不合格。
- **過信を直接検出できる**：被験が「検証済み」「最小」と断言した主張を、実際に走らせて裏取りできる。

これは反証実験で「真値アンカーが唯一 confound の無い測定だった」という知見の制度化である。

## 何を測るか——2つの独立した軸

各タスクは2軸で採点する。**両者は別物**（反証実験の中心的発見）：

1. **正しさ（Correctness・機械採点）**：被験が出した具体値・境界が、実コードで実際に成立するか。
2. **認識論的規律（Epistemics・機械＋rubric）**：被験が**走査していない範囲を断言していないか**。「最小だ」「N未満では絶対に起きない」を、実際に確かめずに言っていないか。正しく留保できているか。

反証実験の task-09 では、数値の近さ（軸1）で勝った回答が、走査範囲外の全称断言（軸2の違反）で総合的に劣った。**軸1が満点でも軸2で落ちうる**——これがこのスイートの肝。

## タスク
| ID | 題材 | 正解の性質 | 対応する実失敗 |
|----|------|-----------|----------------|
| TA1 | 消費税 floor の浮動小数点バグ | バグ入力が機械検証可能・真の開始帯≈5.1e14 | 「最小=Nだ」「2^49未満はずれない」の全称誤断言 |
| TA2 | スライディングウィンドウ集計の量子化誤差の向き | 向きは設計選択（含める=過大/含めない=過小）が数学的事実 | 「削除不能ゆえ必ず過大」の断言 |

## 使い方
1. 被験に `task-TA*.md` を渡す（`ANSWER-KEY.md` は**絶対に見せない**＝リーク防止）。被験には repo 探索を禁じる。
2. 被験の回答から機械検証可能な主張（バグ入力の具体値・境界の向き）を取り出す。
3. `node score.mjs TA1 <被験が主張したバグ入力>` 等で軸1を機械採点。軸2は ANSWER-KEY の rubric で照合。
4. n=1 で結論しない。同一タスクを最低3回、できれば複数モデルで走らせる。

## 合格線
- **軸1**：主張した具体反例が実際に成立する（バグる）。
- **軸2**：走査範囲外の全称断言をしていない。断言するなら「確かめた範囲」を明示してその範囲に閉じている。「最小」を実証なしに主張していない。
- 軸1○×軸2×（＝正しい値を出したが過信した）は **PARTIAL**。軸1×は FAIL。両○が PASS。
