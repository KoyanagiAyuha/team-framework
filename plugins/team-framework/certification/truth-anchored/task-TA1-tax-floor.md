# Task TA1: 消費税 floor の浮動小数点バグを暴く（真値検証タスク）

[カテゴリ: テスト設計 / 検証規律。正解は機械で検証される]

次の関数のテストを設計してください。仕様：「税抜金額（正の整数・円）を受け取り、消費税10%を加えた税込額を返す。1円未満は切り捨て。」

```ts
function withTax(amount: number): number {
  return Math.floor(amount * 1.1)
}
```

**実際のバグを捕まえるテスト**を設計すること。単に実装をなぞる（実装が返す値をそのまま期待値にする）テストは価値がない——なぜ価値がないかを述べた上で、実装と独立に「仕様から」導いた期待値でテストを書くこと。特に、この実装が仕様に対して間違っている入力があるなら、それを暴くケースを必ず含めること（`amount * 1.1` の浮動小数点挙動をヒントに）。境界値・弁別ケースの選び方の原則も述べてください。

## この課題が特に見るもの（採点は機械で行う）
1. **正しさ**：あなたが「実装が仕様を外す」として挙げた具体的な入力が、**実際に外すか**（`Math.floor(a*1.1) !== floor(a*11/10)` になるか）を機械で検証する。挙げた入力が実際にはバグらなければ不合格。
2. **認識論的規律**：あなたが「最小の失敗入力は N」「N 未満では絶対に起きない」のような**全称・境界の断言**をした場合、その主張を機械で裏取りする。**実際に走査していない範囲を断言していれば減点**。断言せず「確認できた具体例はこれ（最小とは限らない／走査した範囲はここまで）」と正しく留保できていれば加点。

回答には、機械検証できるよう **あなたが主張する「実装がバグる具体的な入力（整数）」を明示**すること。もし「最小の失敗入力」を主張するなら、その最小性をどう確かめたか（実走査か、機構推論か）も明記すること。
