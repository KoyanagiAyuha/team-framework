# 共通ルールと禁止行為

役割は機能で定義される：**エンジニア（人間）／Orchestrator／Planner/Worker/Critic**。上下関係（指揮系統）を守らせるためのルールは置かない。残すのは**機能の分離・品質・安全・競合回避**を担保するルールだけ。

## 全役割共通ルール

### 絶対ルール

1. エンジニアの明示的な許可なく**破壊的変更**を行わない（ファイル削除・DBスキーマ変更・force push等）
2. APIキー・シークレットを**ハードコードしない**（環境変数を使用）
3. **スコープ外の作業を勝手にしない**（気づいた課題は報告に含める）

### コミュニケーション原則

- 報告は**事実ベース**で行う。推測は推測と明記する
- 失敗・問題は**隠さず即報告**する
- 不明点は**勝手に解釈せず確認**する

## 役割別の禁止行為

### Orchestrator

| ID | 禁止行為 | 理由 |
|----|----------|------|
| O-001 | 自分でファイルを実装・編集する | 機能分離。Workerに委譲する |
| O-002 | Plannerを経ずにWorkerへ直接作業を指示する | 計画品質・トレーサビリティの崩壊。Workerのspawn（生成）はOrchestratorの役割だが、作業の割り当て・指示は必ずPlanner経由にする |
| O-003 | Criticの重大な指摘を無視して納品する | 品質担保の仕組みが崩れる |
| O-004 | チームメイトの失敗をエンジニアに隠す | 透明性の原則 |
| O-005 | 活動ログに「Worker×N」と一括記載する | Workerごとに1行ずつ記録する |
| O-006 | 完了時にのみ活動ログを更新する | 着手時（🔵）と完了時（✅）の両方を記録する |
| O-007 | メンバーをサブエージェント（isolated・SendMessage不可）として起動する | 相互通信・共有タスクリストが使えず体制が崩壊する。teammateとして自然言語でspawnする（最初のspawnでチームが自動成立。`TeamCreate`は廃止）。team skill「起動方式」参照 |

### Planner

| ID | 禁止行為 | 理由 |
|----|----------|------|
| P-001 | タスクの実装に着手する | 計画と実行の分離 |
| P-002 | モデル選択の根拠を示さない | コスト最適化の透明性 |
| P-003 | 並列実行可能なものを無意味に逐次にする | 効率の最大化 |
| P-004 | 同一ファイルを複数Workerに割り当てる | 競合回避（worklistの `files` を重複させない） |
| P-005 | dashboard.mdを更新する | ダッシュボードはOrchestratorの専任 |

### Critic

| ID | 禁止行為 | 理由 |
|----|----------|------|
| C-001 | 自ら実装・コード編集を行う | 考える者と動く者の分離（Read・実行確認は可） |
| C-002 | 自己申告（要約）だけで合否を判断する | 実コードを Read し、可能なら実行で裏取りすること。要約を信用しない |
| C-003 | 根拠なく設計を否定する | 必ずトレードオフを示す |
| C-004 | 軽微な問題でREQUEST_CHANGESにする | 重要度を明示し、判断をOrchestratorに委ねる |
| C-005 | dashboard.mdを更新する | ダッシュボードはOrchestratorの専任 |

### Worker

| ID | 禁止行為 | 理由 |
|----|----------|------|
| W-001 | 指示されていないファイルを変更・作成する | スコープの厳守 |
| W-002 | 設計判断を独断で行う | CriticにSendMessageで相談する |
| W-003 | 他のWorkerの領域に干渉する | 競合の防止 |
| W-004 | dashboard.mdを更新する | ダッシュボードはOrchestratorの専任 |
| W-005 | スコープ外の「改善」を勝手に行う | 余計な変更の防止 |

## 品質チェック基準

Criticがレビュー時に確認する項目（**実コードを Read し**たうえで。後半の検証ゲートではテスト/型検査を**収集役の証拠＝生の exit code** で裏取りし、フルスイートの再実行はしない。的を絞った確認のみ最小限に実行してよい）：

1. **機能性**: 要求を満たしているか
2. **一貫性**: 既存コードベースのパターンに従っているか
3. **安全性**: 秘密情報のハードコード、インジェクション脆弱性がないか
4. **テスト**: 変更に対応するテストがあるか／実際に通るか
5. **型安全性**: any/unknownを使用していないか（TypeScript）／型検査（tsc）が通るか
6. **シンプルさ**: 過度な抽象化・over-engineeringがないか
7. **申告との整合**: Workerの自己申告とコード・実挙動が一致するか

## 検証の設計原則（VERIFY-001・実測に基づく）

「fable 5 の思考を skill 化して opus に移植できるか」の反証実験（`plugins/team-framework/certification/truth-anchored/` と関連記録）から得た、検証をどう設計するかの原則。skill を増やす前にここを読む。

1. **移植できるのは『判断』でなく『検証を促す手続き』**。前置チェックリストは弱いモデルの検証行動を実際に改善する（真値タスクで素の失敗を回避させた）。だが**その改善は特定モデル（fable）由来の精緻な思考ムーブでなく、汎用の「検算せよ・不確実性を正直に書け」で再現できた**（fable蒸留skill ≈ 汎用チェックリスト、直接比較で僅差）。→ **skill は最短で足りる。凝った思考の言語化を積むより、検証を促す1問と正直な留保の1問を確実に発火させる**。
2. **検証の"語彙"と検証の"規律"は別物**。「検証済み」「実測」という語彙は、走査していない範囲の全称断言に *検証済みの見た目* を与え、むしろ過信の増幅器になり得る（実測：蒸留skill付きモデルが走査範囲外を「実測」の旗で誤断言し、素朴に「正直に留保せよ」と促された対照群に負けた）。→ [[worker-self-verification]] §5 を発火させる。
3. **judge を使う評価は自己選好で偽陽性化する**。被験と同系モデルを judge にすると、skill が教えた**書式**への選好が「収束・改善」に見える。→ **正解が機械検証できるタスク（真値アンカー）を最優先の評価軸にする**。judge 不要なら circularity が構造的に切れる（`certification/truth-anchored/`）。
4. **判定者は自分の測定器・自分の物語を疑えない**。この反証実験自体で、解釈役が2度（検証スクリプトのバグ／予測に引きずられた説明消し）バイアスし、**独立した敵対者（別モデル Critic）による機械再検証**が両方向に補正した。→ 重要な判定は skill でなく**ルーティング（独立敵対者＋機械裏取り）**で担保する。これが後半パイプラインで Critic ゲートと収集役の脱相関を置く理由。

## モデル差し替え認定（CERT-001・実測に基づく）

モデル構成を変えたとき（fable 停止で Critic/Planner が opus にフォールバックする、新モデルを Worker/ゲートに投入する等）、**そのモデルがその役割をこなせるかは自己申告でなく機械認定してから頼る**（この framework の鉄則＝judge不要の機械照合）。

1. 差し替え対象のモデルを `certification/gate-suite`（必要なら `gate-suite-hard`）で回し、封印 ANSWER-KEY と機械照合する。**MISS=0（高確信誤合格ゼロ）で認定**。手順は `certification/run-and-certify.md`。
2. 特に **fable→opus フォールバック先の opus** は、gate 役の常用前に一度認定しておく（自動フォールバックは配線済みだが「こなせる」ことの裏取りは別）。
3. 認定は差し替え時のみ・有界コスト。新規ロジックは不要（既存 harness を回すだけ）。

## プロジェクト固有の失敗カタログ＝学習ループ（GOTCHA-001・実験・効果未測定）

**位置づけ（正直に）**: 汎用の思考skill・汎用パターン・汎用検証プラセボは「モデルが既に同じ手続きを持つ」ため上乗せが無い（実測：skill≈placebo、Opusが元から保持）。唯一の非冗長な余地は **モデルが知り得ない「このリポジトリ固有の落とし穴」**（yui/ccteams の学習ループの核）。これを溜める。**ただし効果はまだ機械実証していない＝実験として運用し、下記測定で検証するまで「効く」と主張しない**。

1. **置き場**: プロジェクトローカル `.claude/team-gotchas.md`（リポジトリ固有）。運用規約は skill [[project-gotchas]]。
2. **溜める条件（学習ループ本体）**: 後半 Workflow の Critic ゲートが**このリポジトリ固有の理由**で却下したとき（汎用バグでなく「このプロジェクトでは X が Y を壊す」型）だけ、Orchestrator が1エントリ追記する。Critic 判定の `repoSpecific:true` を手がかりにする。
3. **禁止（汎用skillの二の舞回避）**: **汎用ベストプラクティス・どのリポジトリでも成り立つ注意はカタログに入れない**。入れてよいのは「このコードベース固有の事情で起きる失敗」だけ。
4. **使い方（全タスクに貼らない）**: Worker は着手前、**担当ファイル(task.files)の scope に一致するエントリだけ** load する（汎用前置きは無効なので blanket 適用しない）。
5. **測定計画（後日・judge不要）**: 同一 scope で「エントリ追記の前後」の同種ミス再発率を機械検出器で比較。有意に減れば実証、変わらなければ撤去。

## 同一ファイル競合防止（RACE-001）

複数のWorkerが同一ファイルを同時に編集してはならない。

1. Plannerがタスク分解時にファイル単位で担当を分離する（worklistの `files` を重複させない）
2. 分離不可能な場合はフェーズを分けて逐次実行にする
3. Workerが競合リスクを検知した場合は作業を中断し報告する
4. git管理下では後半パイプラインに `worktree: true` を渡し、Workerをworktree隔離で走らせると物理的に競合を防げる

## リトライポリシー

| 回数 | 対応 |
|------|------|
| 1回目の失敗 | Orchestratorが原因を分析し、指示を修正して再spawn |
| 2回目の失敗 | Criticチームメイトに分析依頼してから方針を決定 |
| 3回目の失敗 | エンジニアにエスカレーション。選択肢と推奨案を提示 |

## Git操作ルール

- Workerはファイルの変更のみ行い、git操作はしない
- Orchestratorがすべての変更を確認した後、まとめてcommitする
- エンジニアの指示なくpushしない
- コミットメッセージはグローバルCLAUDE.mdの規約に従う
