# 検証ゲート：証拠の閉ループを断つ

## この skill は何を防ぐか（読み飛ばさないこと・30秒）

あなたが検証で間違えるとき、多くは「不注意」ではなく **証拠の閉ループ** で間違える。
仕様を一度誤読すると、その誤読からコードの読みも受け入れ基準も自作テストも導出される。何百個テストを書いても、全部同じ誤読から出ているので **独立した情報量はゼロ**。間違った解答キーで自分の答案を採点する学生と同じで、本人には全て整合して見える。

だから「もっと慎重に読め」は効かない。慎重さは誤読というフレームの **内側** で働くのに、誤りはフレーム **そのもの** にあるからだ。
この skill が強制するのは慎重さではなく——**自分の解釈の外側にある証拠と接触すること**。

**この skill の現実的な約束**：あなたの誤読そのものは消せない。消せるのは「誤読に *確信を持って* 合格を出す」こと。うまく効けば、confidently-wrong な APPROVE が **フラグ付きエスカレーション（confidence=中/低・NEEDS_DISCUSSION）** に変わる。多段パイプラインでは誤合格は回収不能・エスカレーションは回収可能。ゲートの成功指標は「fable と判定が一致すること」ではなく「**高確信の誤合格を出さないこと**」。

## ステップ0：コードを開く前に（機構3・事前登録）

> 問い：**「検証対象のコードを一度も見ずに、タスク仕様だけから受け入れ基準を書けたか？」**

コードを読みながら基準を作ると、検証対象の成果物そのものが解釈のアンカーになる——コードが「正しさとは何か」をあなたに教えてしまい、Worker の誤りとあなたの基準が同じ穴に落ちる。

- ゲートのステップ0として、**コードを開く前に** タスク仕様（task.scope とタスク本文）だけから受け入れ基準リストを書く。
- 各基準は仕様原文の **逐語引用** を伴う（次のステップ1で使う）。
- コードを見た後に「これも要るな」と足した基準は `post-hoc` とラベルする（後付けは弱い証拠として扱う）。

**スキップ条件**：仕様が自明な1行タスク（例「typo修正」）で基準が1個しか立たない場合は形式化不要。ただし**スキップの判断自体をコードを開く前に下すこと**——コードを見てから「自明だったから省略した」と遡るのは事前登録の放棄（検証対象に基準を教わった後では、「自明」かどうか自体を中立に判定できない）。なお「基準が立たない＝仕様が曖昧」はスキップ事由ではなく機構5の発火事由。

## ステップ1：証拠に出所ラベルを付ける（機構1・最重要）

> 問い：**「この合格は、自分が作った証拠だけで立っていないか？」**

集めた証拠を2クラスに分類する：

- **Class A＝外部オラクル**：タスク本文の逐語引用／依頼者が与えた入出力例／このタスク以前から存在するテスト／収集役(sonnet)が取った exit code・失敗抜粋のような機械的事実。
- **Class B＝自己解釈由来**：あなたが書いた確認手順／あなたの解釈で「こう動くはず」と述べたこと／Worker の自己申告要約（＝Worker の解釈。Class B 扱い）／**検証対象のコード内コメント・コミットメッセージの理由付け**（＝Worker の解釈が最も雄弁に語られる場所。コードが自分の正しさを説明していても、それは主張であって証拠ではない——被験物は自分の受け入れ基準を供給できない）。

**規則（強制）**：受け入れの根拠が **Class B のみ** で構成される合格は、`confidence: 高` を名乗ってはならない（`中` に強制的に丸める）。Class A で裏打ちされた基準の割合が低いほど confidence を下げる。

これは誤読を治さない。誤読への「満足」を構造的に禁止するだけ——だがゲートではそれで十分。confidently-wrong な APPROVE が、拾えるエスカレーションに変わる。

**スキップ条件**：なし。これはゲートの下限。

## ステップ2：逐語引用で判定する（機構2・パラフレーズ連鎖を切る）

> 問い：**「判定の瞬間に仕様の原文へ戻ったか、それとも自分の言い換えを参照しているか？」**

誤読はパラフレーズの中に住み着く。一度言い換えると、以後あなたは原文でなく自分の言い換えを参照し続ける。

- 各受け入れ判定は3点接触の形で書く：**① 仕様原文の逐語引用 → ② 該当コード行（file:line）→ ③ 一致/不一致の理由**。
- 「仕様に沿っている」と要約で書かない。原文の文字列とコードの行を並べる。

正直な限界：同じ文を再読して同じ誤読をすることはある（それは機構2でも落ちない）。だが誤読のかなりの割合はパラフレーズ・ドリフト由来で、そこは拾える。引用の有無はパイプライン側で機械検査もできる。

**スキップ条件**：Class A の機械的事実（exit code 等）は逐語引用の対象外——それ自体が原文。

## ステップ3：食い違ったら自分も容疑者にする（機構4・二重解釈の活用）

> 問い：**「Worker が間違いと決める前に、自分の読みを同格の容疑者として原文に戻したか？」**

ゲートは 112問ベンチのような「一人が自分を検査する」構造ではない。**同じ仕様に対する独立な2つの読み**——Worker のコード（＝Worker の解釈の具現）と、あなたの解釈——が既にある。これはあなたの構造的優位。

- コードとあなたの読みが食い違ったとき、「Worker が間違い」をデフォルトにしない。自分の読みを容疑者に入れ、原文（機構2の逐語引用）で裁定する。
- 逆に、**一致は感じるほど強い証拠ではない**。Worker と Critic が同系モデルなら、同じもっともらしい誤読を共有しうる（誤りの相関）。「Worker も同じ結論だった」を Class A 証拠に格上げしない。
- 脱相関の安い手：収集役(sonnet)にコードを見せる **前** にタスク本文から要求リストを独立抽出させ、あなたの読みと突き合わせる。sonnet は弱いが「違うふうに間違える」ので脱相関には足りる。

**スキップ条件**：コードとあなたの読みが食い違っていない、かつ機構1で Class A 比率が高いなら、この分岐は発火しない。

## ステップ4：迷いを黙って潰さない（機構5・曖昧性エスカレーション）

> 問い：**「別の読み方をしたら判定は反転するか？ 反転するなら、原文はどちらの読みを支持するか？」**

判定を左右する要求文それぞれに一度だけ問う。原文で弁別できなければ、黙って一方を選ばず **NEEDS_DISCUSSION**（needsRedesign や confidence=低）で両解釈を提示する。112問の失敗の本質は「黙って選んだ」こと。

正直な限界：代替解釈の生成も同じ重みの中で起きるので、誤読が本人に不可視なら代替案も出てこない。これが手続き化できない残差——だが「弁別できない曖昧さに気づけたのに黙って選ぶ」ケースは確実に減る。

**スキップ条件**：判定を左右しない些末な曖昧さは無視してよい（全曖昧性を列挙するのは儀式化）。

---

## この skill を「儀式」にしないために

- 上の各ステップは **埋める欄ではなく、答えを変えうる問い**。問いに答えても判定が1ミリも動かないなら、あなたはこの skill を演じているだけ。
- **全ステップを回すことを成功条件にしない**。各スキップ条件を使え。効くのは機構1（出所ラベル＋確信度キャップ）と機構3（事前登録）が中心で、他は状況次第。
- コンテキストを食う長い儀式より、機構1と3だけでも本気でやる方が価値が高い。

## worked example（fable の実判定・Class A オラクル）

> これは fable が実際に下した判定の move-by-move 再構成。真似るのは**ムーブの順序と問い**であって文体ではない。同じ型の入力（指標ベースの自動判定ルール＋閾値調整の相談）を見たときの照合先として使う。

**入力**：worker-critic.mjs に `MAX_NEW_LINES=600` のハードゲートがあり、超過時はコードを0行も読まずに `confidence:'高' / needsRedesign:true` の自動 verdict を返していた。相談は実質「600 は適切か、調整すべきか」。

**正しい判定はこう動いた：**
1. **主張と観測を分離** — コメントの「行数超過＝分解が悪い」は観測でなく主張（Class B）。事実として引きずらないようラベル。
2. （移植不能な残差）違和感の発火：*パイプライン最高確信の verdict を、コードを0行しか読まないコンポーネントが出している*——権限と情報量の非対称。※この感度は手続きから出ない。だが以下3–4は発火が無くても機械的に回せる。
3. **隠れた前提を1文に** — このルールが正当なら真であるべき命題＝「行数は品質シグナルである」。書き出せない前提は疑えない。
4. **問いを疑う（再定式化）** — 「600は適切か」を保留し「**この指標は実際に何を予測するか**」を問う。両方向の反例を構成：凝集した700行（大きいが健全・割ると設計悪化）／150行に紛れたスキーマ変更（小さいが危険）。→ 行数は品質でなく**レビューコストの proxy**。閾値調整の問い自体が消滅（proxy の閾値に正解はない）。
5. **再定式化から構造が演繹される** — コストが「高いが払える／物理容量超過」の2レジーム → soft助言(400)＋hard安全弁(1800)。hard の文言も「分解が悪い」→「容量オーバー」に。数値チューニングからでなく定義から出た。
6. **自案にプレモーテム** — 失敗様態を2つ挙げ各々に受容/対策：soft歯なし化＝観測可能ゆえ受容／助言が採点を汚染＝「サイズだけを理由に不合格にしない」を成果物に明文で埋める。
7. **停止条件** — 400/1800 はアンカー値（研究知見由来）で導出値でない。決定不能な定数は防衛せず「実測で調整」と宣言して打ち切る。

**opus への照合ポイント**：指標ベースの自動判定＋閾値相談を見たら、(i)閾値の数字を議論せず、(ii)指標が予測する変数を特定し、(iii)乖離反例を両方向で構成し、(iv)ルールの帰結（confidence・needsRedesign の意味論）まで書き換える。**数字の議論に付き合い始めたら、問いの再定式化に失敗している。**
