# 分解の検査：悪い worklist を却下する

## なぜ「生成」でなく「却下」か（20秒）

良い分解の中核＝「凝集した自然な継ぎ目を見つける」感覚は重みの中にあり、チェックリストにできない。だが **悪い分解には共通の型がある**。ここで拾えるのはその型——「これは悪い」を確実に棄却できれば、下流（Worker 並列・Critic ゲート）の事故の多くは未然に消える。

分解を確定する前に、各タスクと worklist 全体に対して以下を問う。**1つでも "いいえ" が残るなら、その分解はまだ配らない。**

---

## タスク単位の検査（各 task ごと）

> **① 独立に検証可能か** — このタスクの成否を、他タスクの成果を待たずに Critic ゲートが判定できるか？ できないなら、検証境界がタスク境界とずれている。
>
> **② 依存は deps に明示されているか** — 暗黙に「t2 は t1 の後」と思っているのに `deps` が空、はよくある穴。頭の中の順序を deps に落としたか？
>
> **③ 1パスでレビューできる凝集か** — Critic が1回で読み切れる量・意味のまとまりか（目安 SOFT 400行／安全弁 HARD 1800行）。ただし **行数合わせで凝集した単位を割っていないか**（数字のために自然な継ぎ目でない所で切ると、分割しても総レビューコストは減らず deps だけ増える）。
>
> **④ 失敗時に切り戻せる単位か** — このタスクだけ REQUEST_CHANGES で差し戻したとき、他タスクを巻き込まず再実行できるか？
>
> **⑤ タスク文は受け手だけで一意に解釈できるか** — Worker はあなたの頭の中の文脈を持たない。「上記の方針で」「例の形式に合わせて」等、Planner の文脈を前提にした語が残っていないか？ **別の読み方をしたら成果物が変わる文**が1つでもあれば、タスク文の段階で潰す（Critic の曖昧性エスカレーションの上流版——ここで潰す方が1往復安い）。判定に迷ったら、成功基準を1行足して曖昧さを消す。

## worklist 全体の検査

> **⑥ files がタスク間で重複していないか** — 同一ファイルを2タスクが触ると並列で競合する。files 集合は互いに素か？（worktree:true でも論理的重複は避ける）
>
> **⑦ モデル割当は最小か** — opus Worker は3類型（大規模リファクタ／難読バグ／複雑アルゴリズム）だけ。「念のため opus」になっていないか？ effort を既定から上げていないか？
>
> **⑧ タスク数は最小か** — 段（タスク）を「組織のリアルさ」や「丁寧さ」で増やしていないか。コンテキスト隔離が正当化する分割だけ残す。

---

## 却下したらどうするか

- ①③が "いいえ"：検証境界／凝集の継ぎ目でタスクを引き直す（行数でなく意味で）。
- ②が "いいえ"：deps を補う。
- ④が "いいえ"：他タスクと密結合しているサイン。統合するか、共有部分を先行タスクに括り出す。
- ⑤が "いいえ"：タスク文を書き直す（受け手だけで一意になるまで。迷ったら成功基準を1行足す）。
- ⑥が "いいえ"：files を割り直すか、同一ファイル群を1タスクに統合。
- ⑦⑧が "いいえ"：格下げ・統合してから配る。

## 儀式化を避ける

- これは「分解のたびに8項目の作文を書く」ためのものではない。**"いいえ" を1つ見つけて配布を止める**のが仕事。全項目 "はい" が自明な軽い分解（1〜2タスク）は、②（deps）と⑥（file重複）だけ確認すれば通してよい——2タスク構成の事故はほぼこの2つ。
- 良い分解が既にできているなら、この検査は数十秒で終わる。時間がかかるのは分解が実際に悪いとき——それがこの skill の狙い。
