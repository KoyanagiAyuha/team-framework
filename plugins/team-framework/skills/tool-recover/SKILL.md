【重要・作業を止めて確認してください】

直前のあなたの出力に、ツール呼び出しが実行されず、`court` や `count`、
`<invoke name="...">` というテキストがそのまま本文に出力されている箇所が
あります。これは tool_use が構造化ブロック（tool_useブロック）として発火せず、
テキストに化けた「壊れたツール呼び出し」の可能性が高い状態です。

補足: $ARGUMENTS

次の手順を、**推測ではなく事実確認ベース**で実行してください。

## 1. 各アクションの実行有無を「実環境の状態」で検証する

直近の各アクション（コマンド実行・ファイル編集・メッセージ送信・タスク登録など）
について、「実際に実行されたか」を、会話履歴の見た目ではなく**実環境の状態**で
検証すること。

- ファイル編集 → 対象ファイルを実際に Read で読み直して反映を確認
- コマンド実行 → 実際の出力・終了コード・生成物の有無を確認
- メッセージ送信（SendMessage等） → 送信結果や受信側の状態を確認
- タスク登録/更新 → 共有タスクリストやdashboard.mdの実状態を確認

会話に `<invoke>` テキストが残っているだけでは「実行された」と見なさないこと。
「実行したつもり」を最大の警戒対象とすること。

## 2. 結果を二分して報告する

検証結果を「**実行が確認できたもの** / **実行が確認できないもの**」に分けて
報告すること。確認できないものは、**未実行として扱う**こと。

## 3. 壊れた出力を手本にしない・闇雲にリトライしない

- 壊れたツール呼び出し（`<invoke>`テキスト等）を見ても、それを**手本に同じ形式を
  再生産しない**こと。
- 同じ内容を**即座にリトライしない**こと。直前の壊れた出力にアンカーして
  決定論的に同じ失敗を繰り返すため。
- 1〜2回試して直らない場合は「**壊れている**」と明言し、作業を続行せず
  エンジニア（ユーザー）に報告すること。回復不能なら `/clear`（Agent Teamsなら
  チーム再作成）で汚染履歴を断つのが確実、と添えること。

## 4. 以降のツール使用ルール

今後ツールを使うときは、前置きの説明文を書かず、**応答の先頭でツールを呼び出す**
こと（説明はツール結果が返ってから書く）。

## 5. （Agent Teams時のみ）teammateへの伝播

補足に `teammate` の語がある、または現在Agent Teamsのチームを稼働中の場合は、
各teammate（Planner・Worker・Critic）にも本検証を SendMessage で伝播し、
それぞれの未実行アクションを同じ基準で洗い出させること。

---

まず **手順1と2の検証結果だけ**を報告してください。
未実行のものへの対応は、その報告を見てからエンジニアが指示します。続行しないこと。
