# 自己検証：自分の誤読でテストを書かない

## この skill が防ぐもの（30秒）

あなたが実装で間違えるとき、多くは「不注意」ではなく **証拠の閉ループ**：仕様を一度誤読すると、その誤読からコードも、確認手順も、テストも導出される。テストが全部緑でも、それは「仕様に合っている」ではなく「**自分の実装と自分のテストが一致している**」を確認しただけ。間違った解答キーで自分の答案を採点する学生と同じで、あなたには全部整合して見える。

「全テスト green です」と報告した瞬間、あなたはこの罠に一番近い。**green は Class B（自作テスト）で立っているか、Class A（仕様が定めた事実）で立っているかを、報告前に一度だけ確かめる。**

---

## 1. 実装を書く前に、受け入れ基準を仕様から事前登録する

> 問い：**「コードを1行も書く前に、仕様だけから『何が正しいか』を書き出したか？」**

先にコードを書くと、コードが「正しさとは何か」をあなたに教えてしまう。順序を逆にする：仕様（依頼文・受け入れ例・制約）から受け入れ基準リストを**実装前に**書く。各基準は仕様原文の**逐語引用**を添える（あなたの言い換えでなく原文）。

**スキップ条件**：仕様が自明で解釈の余地が1つしかないタスク。ただし「自明」の判断は実装を見る前に下すこと。

---

## 2. 依頼者が与えた例は、逐語でテストにする（＝Class A 化）

> 問い：**「依頼文にある受け入れ例・入出力例を、そのままアサートにしたか？」**

依頼者提供の例は最強の Class A 証拠（外部オラクル）。これをテスト化しないのは、手元にある答え合わせを捨てるのと同じ。

- 受け入れ例（表・サンプル入出力）は**1つ残らず逐語でアサートに写す**。`expect(f(1050)).toBe(32)` のように、あなたの計算でなく依頼者の数字で。
- 例に「実装はこの表と完全一致すること」等の条件があれば、全行をテストにする。

> ※ 実データ（第2ラウンド T1）：付与率3%の実装で受け入れ例 `1050円→32pt` を*テストに含めなかった*ため、floor(31.5)=31 の誤りが緑をすり抜けた。例をテスト化していれば源流で落ちていた。

---

## 3. 自作テストには Class B ラベルを貼り、それだけで満足しない

> 問い：**「この green は Class A（仕様の例）で立っているか、Class B（自分が考えたケース）だけで立っているか？」**

あなたが「こういうケースもあるはず」と考えて足したテストは Class B——あなたの解釈の産物であり、誤読していれば一緒に間違っている。

- テストを Class A（仕様の例由来）と Class B（自作）に仕分ける。
- **Class B のみで構成された green は「仕様適合の証拠」として報告しない。** 「自作テストは通ったが、仕様の受け入れ例◯件は未検証」と正直に書く。

---

## 4. 弁別ケース（境界）を狙い撃つ。楽なケースで緑にしない

> 問い：**「自分の読みと『別のありうる読み』で答えが割れるケースを、テストしたか？」**

慣習で引っ張られる誤読は、楽なケースでは露見しない。**あなたの解釈と代替解釈が同じ答えになるケースを何個通しても、独立情報量はゼロ。**

- 仕様の中で解釈が割れうる点（丸め＝切捨/四捨五入、順序＝昇順/降順、境界＝以上/超、週開始＝日曜/月曜、日付＝UTC/ローカルTZ、0/1-index、「直近N件」の並び）を洗い出す。
- **各点で、2つの読みが違う答えを出す入力**を作ってテストする。あなたの読みが正しいなら、依頼者の例がそれを支持するはず（＝2に戻る）。

> ※ 実データ（第2ラウンド T3/T4/T6）：並び順・週開始曜日・UTC/JST境界の各欠陥は、いずれも自己テストが「2つの読みで答えが割れないケース」だけを通して緑にしていた。弁別ケースを1つ足せば全て源流で落ちた。

---

## 5. 走査した範囲を超えた主張をしない（検証の語彙で過信を語らない）

> 問い：**「『検証済み／実測』と書いたその主張は、実際に走らせた入力の範囲に収まっているか？ それとも範囲外まで一般化した全称主張になっていないか？」**

閉ループのもう一つの出方は、**検証の"語彙"だけを借りて、実際の検証範囲を超えて断言する**こと。「実測した」「機械検証済み」と書くと、その旗の下に置いた主張はすべて Class A に見える——だが *走査した範囲* が Class A なのであって、*そこから外挿した全称命題* は Class B（あなたの推論）のままだ。

- 「X は決して起きない」「N 未満では必ず正しい」のような**全称・境界の断言**をするときは、**その範囲を実際に走査したか**を確かめる。していないなら「1〜10⁸ の範囲では観測されなかった（それ以上は未走査）」と、検証した範囲を明示して主張を**その範囲に閉じる**。
- 機構推論で境界を出したなら、それは Class A（観測）でなく Class B（推論）とラベルする。推論した境界の**すぐ外側**に反例が無いか、最低1点だけでも実際に走らせて裏を取る（安い弁別観測）。
- 「厳密な最小値だ」「ここが境界だ」と言い切れないなら、**言い切らない**。「確認できた具体例はこれ（ただし最小とは限らない）」で止める。回帰テストには*確認済みの具体反例*を固定すれば足り、最小性の断言は要らない。

> ※ 実データ（汎用思考skillの反証実験・task-09）：fable蒸留skillを積んだ opus は、全数走査したと**報告した**範囲は 1〜10⁸ までなのに、（2⁴⁹ を挙げて）「それ未満では物理的にずれようがない」と「すべて実測」の旗の下で断言した。だが 2⁴⁹ 未満に実バグ入力が多数実在し（機械確認済み・帯 [5.12e14, 2⁴⁹) のサンプルで約1割がバグ）、この全称否定は偽だった。実際、断言した最小値の**わずか10下**の入力が既にバグる——「境界のすぐ外側を1点走らせる」処方1つで即座に反証できた。同じ問題を、「不確実性を正直に書け」を含む汎用チェックリストを与えられた対照群は「厳密な最小値ではない」と正しく留保して回避した。**検証の語彙を持つことと、検証の規律を持つことは別物**——語彙は、範囲外の断言に *検証済みの見た目* を与えてしまう分、むしろ過信の増幅器になり得る。

---

## 儀式化を避ける

- これは「毎回4項目の作文を書く」ためではない。**green を報告する前に、それが Class A で裏打ちされているかを1回問う**のが本体。
- 効くのは 1（事前登録）と 2（例の逐語テスト化）。3・4 は仕様に解釈の割れ目があるときだけ、5 は「決して/必ず/最小/境界」のような全称・境界の断言をするときだけ発火。該当しない単純タスクでは飛ばしてよい。
- 完了報告に一言添える：「受け入れ例◯件を逐語テスト化、うち弁別ケース△件を確認。未検証は□」。この一文が書けないなら、まだ閉ループの中にいる。
